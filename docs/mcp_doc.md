# MCP 服务 🔧(Model Context Protocol) 服务使用说明文档

**文档更新时间：2026年4月16日**

**Anywhere Desktop** 对 **MCP (Model Context Protocol)** 提供了深度支持。借助 MCP，AI 不再只是聊天模型，而是能够在桌面端真实调用本地文件、执行命令、运行脚本、联网搜索、调度任务，甚至与其他 Agent 协作的工作执行体。

本文档将说明如何在桌面端配置和使用 MCP 服务，把 Anywhere Desktop 升级为真正的本地 AI 自动化工作站。

---

## 1. 前置准备 (必读)

在使用 MCP 之前，请先确认当前电脑满足以下环境要求。

### 1.1 环境安装

绝大多数外部 MCP 服务依赖 Python 或 Node.js 运行。

* **Python 环境**：如果您要使用 Python 相关 MCP 或内置 Python 执行器，请先安装 Python 或 Anaconda。
* **Node.js 环境**：若您要运行基于 `npx` / JavaScript 的 MCP 服务，请先安装 Node.js。
* **Git**：部分第三方 MCP 或技能流程会依赖 Git。

**⚠️ 重要提示：**
安装或修改环境变量后，建议完全退出并重新启动 Anywhere Desktop；必要时可重启电脑，以确保主进程能读取到最新 PATH。

### 1.2 模型要求

MCP 的核心依赖是 **Function Calling（工具调用）**。

* **推荐模型**：Claude、GPT-4o、DeepSeek-V3 / R1 等工具调用能力强的模型。
* **不推荐模型**：工具调用能力弱、格式不稳定、无法稳定输出 JSON 参数的模型。

---

## 2. 内置 MCP 服务 (开箱即用)

Anywhere Desktop 预置了一组桌面端高频使用的内置 MCP 服务。建议默认保持启用，它们构成了桌面自动化的基础能力。

* **🤖 Super-Agent**：
  * 用于多 Agent 协作；
  * 支持后台子智能体（Sub-Agent）；
  * 支持调度其他独立窗口中的 Agent。
* **📂 File Operations**：
  * 读、写、精确替换、正则替换、插入、搜索文件内容；
  * 对代码 / Markdown / 文本文档处理尤其高效。
* **🐍 Python Executor**：
  * 自动扫描本机 Python 环境；
  * 允许 AI 生成并运行 Python 脚本。
* **💻 Shell Executor**：
  * 在 Windows 下默认执行 PowerShell；
  * 支持后台长任务与日志读取。
* **🌐 Web Toolkit**：
  * 提供联网搜索、网页抓取与远程内容读取能力。
* **⏰ Task Manager**：
  * 允许 AI 创建、更新和管理桌面端定时任务。
* **🕒 Time Service**：
  * 提供当前时间与时区时间读取能力。

## 3. 添加自定义 MCP 服务

除了内置服务外，您也可以接入社区中的第三方 MCP 服务。

### 3.1 配置类型 (Type)

#### Stdio (标准输入输出)

这是最常见的本地 MCP 形式，桌面端会以子进程方式启动它。

* **命令 (Command)**：如 `npx`、`python`、`uv`、`docker`。
* **参数 (Args)**：每行一个参数。
* **环境变量 (Env)**：用于填写 API Key 或其它运行所需环境变量。

![img](..\image\MCP配置界面.png)

#### SSE / HTTP

用于连接本地或远程运行中的 HTTP MCP 服务。

* **URL**：填写完整地址，例如 `http://localhost:3000/sse` 或 `http://127.0.0.1:3000/mcp`。

### 3.2 关键设置

* **持久连接 (Persistent Connection)**：
  * **关闭**：按需启动、用完即断，节省资源；
  * **开启**：服务常驻后台，适合启动很慢或需要保持状态的 MCP。

对于桌面端来说，持久连接适合数据库连接、长生命周期 REPL、索引服务等场景。

## 4. 在对话中使用 MCP

### 4.1 启用服务与工具

在独立对话窗口中，您可以通过输入区下方的 MCP 面板，选择本次对话允许调用的服务与工具。

* 支持按服务粒度启停；
* 也支持关闭某个服务下的特定工具，减少误调用风险。

### 4.2 AI 如何调用

启用后，您不必手动描述工具细节，只需自然表达目标即可。

例如：

* *用户*：“帮我在项目里找一下所有用到 `handleSubmit` 的地方。”
* *AI (后台)*：调用 `grep_search`。
* *AI (回复)*：给出命中文件与行号。

再例如：

* *用户*：“帮我开一个专门写文案的 Agent，把这段内容润色一下。”
* *AI (后台)*：可调用 `summon_agent` / `sub_agent` 等能力完成任务。

### 4.3 审批机制 (安全锁)

为了防止 AI 在桌面端执行危险操作，Anywhere Desktop 提供审批机制：

* **手动批准**：默认推荐。每次工具调用前由您确认。
* **自动批准**：适合高度信任的自动化场景，但请谨慎开启。

涉及删除文件、覆盖内容、执行系统命令等操作时，建议保留人工确认。

## 5. 常见问题 (FAQ)

**Q1: 添加了 Stdio 服务，但连接测试失败怎么办？**

> **A:**
>
> 1. 检查命令是否存在于系统 PATH；
> 2. 若是 `npx`，检查 Node.js 与网络环境；
> 3. 若是 Python，必要时使用绝对路径，如 `C:\Python311\python.exe`；
> 4. 修改环境后建议重启桌面端。

**Q2: 为什么持久连接数量满了？**

> **A:** 持久连接会持续占用系统资源。若同时开启太多第三方服务，可能出现资源占满或连接上限问题。建议关闭长期不用的持久服务。

**Q3: 为什么 AI 总是乱调工具或参数格式错误？**

> **A:** 这通常与模型的工具调用能力有关。请尽量选用支持函数调用且稳定输出结构化参数的模型。

**Q4: 修改 MCP 环境变量后需要重启应用吗？**

> **A:** 一般建议刷新或重连对应服务；若 PATH 等系统环境变动较大，建议直接重启桌面端。

**Q5: MCP 和 Skill 有什么区别？**

> **A:**
>
> * **MCP** 是工具，是 AI 的“手”；
> * **Skill** 是流程与规范，是 AI 的“SOP / 方法论”；
> * Skill 往往会指导 AI 正确调用 MCP 来完成具体工作。
